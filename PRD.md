# PRD: 반도체 시료 생산주문관리 시스템

## 1. 배경

가상의 반도체 회사 "S-Semi"는 연구소, 팹리스(Fabless) 업체, 대학 연구실 등에 반도체 시료(Sample)를
생산·납품한다. 시료는 주문이 들어오면 웨이퍼 공정 설비를 통해 제작되고, 검수를 거쳐 고객에게
출고된다.

주문량이 급증하면서 엑셀/메모장 기반 관리로는 아래와 같은 문제가 발생했다.

- 주문 처리 여부를 파악하기 어렵다.
- 공정 예약 후 완성 시점을 알 수 없다.
- 재고가 충분한데도 불필요한 추가 공정이 돌아간다.

이를 해결하기 위해 콘솔 기반 "반도체 시료 생산주문관리 시스템"을 구축한다.

## 2. 목표

- 시료 등록·조회·검색을 체계화한다.
- 주문 접수부터 승인/거절, 생산, 출고까지의 전체 흐름을 추적 가능하게 만든다.
- 재고와 생산 현황을 담당자가 한눈에 파악할 수 있도록 모니터링을 제공한다.
- 재고가 충분한 주문은 즉시 출고 대기로, 부족한 주문만 생산 라인에서 자동 생산되도록 하여
  불필요한 공정 낭비를 없앤다.

## 3. 사용자 및 역할

| 역할 | 설명 |
|------|------|
| 고객 | 필요한 시료를 이메일 등으로 요청하는 시료 요청자 (시스템 외부, 직접 조작하지 않음) |
| 주문 담당자 | 고객 요청에 맞춰 주문서를 작성·관리 (시료 예약 생성) |
| 생산 담당자 | 개발 시료 등록, 주문 수신 후 승인/거절, 생산·출고 처리 |

시스템은 단일 콘솔 애플리케이션으로 동작하며, 위 역할은 사용자 계정 구분 없이 동일한 콘솔을
공유해서 사용한다고 가정한다 (로그인/권한 분리는 범위 밖).

## 4. 기능 요구사항

### 4.1 메인 메뉴
- 기능별 선택 화면을 표시한다.
- 등록 시료 종수, 총 재고, 전체 주문 수, 생산라인 대기 건수 등 전체 요약 정보를 표시한다.
- 하위 메뉴: 시료 관리 / 시료 주문 / 주문 승인·거절 / 모니터링 / 생산 라인 조회 / 출고 처리

### 4.2 시료 관리
- **시료 등록**: 시료 ID, 이름, 평균 생산시간, 수율을 입력받아 신규 시료를 추가한다.
- **시료 조회**: 등록된 모든 시료 목록과 현재 재고 수량을 함께 표시한다.
- **시료 검색**: 이름 등 속성으로 특정 시료를 검색한다.
- 시스템에 등록된 시료만 주문 가능하다.

### 4.3 시료 주문 (예약)
- 고객이 원하는 시료 ID, 고객명, 주문 수량을 입력받아 주문을 생성한다.
- 생성된 주문은 `RESERVED` 상태를 가진다.

### 4.4 주문 승인/거절
- `RESERVED` 상태의 주문 목록을 표시한다.
- **승인**: 재고 상황에 따라 자동으로 분기한다.
  - 재고 충분 → 주문을 즉시 `CONFIRMED`로 전환한다.
  - 재고 부족 → 생산 라인에 자동 등록하고 주문을 `PRODUCING`으로 전환한다.
- **거절**: 즉시 `REJECTED`로 전환한다. `REJECTED`는 정상 흐름 밖의 상태이며 이후 모든 모니터링에서
  제외된다.

### 4.5 모니터링
- **주문량 확인**: 상태별(`RESERVED`/`CONFIRMED`/`PRODUCING`/`RELEASE`) 주문 수를 표시한다.
  `REJECTED`는 집계에서 제외한다.
- **재고량 확인**: 시료별 현재 재고 수량과, 주문 대비 재고 상태(여유/부족/고갈)를 표시한다.
  - 여유: 주문 대비 재고 충분
  - 부족: 재고는 있으나 주문 대비 부족
  - 고갈: 재고 수량이 0

### 4.6 생산 라인
- 단일 생산 라인이 시료를 하나씩 생산하며, 주문이 들어온 시료에 대해서만 생산한다.
- 생산 대기열은 **FIFO**(선입선출)로 스케줄링한다.
- 실 생산량 = `ceil(부족분 / 수율)`, 부족분 = `주문 수량 − 현재 재고`
- 총 생산 시간 = `평균 생산시간 × 실 생산량`
- 생산 완료 시 재고에 실 생산량을 반영하고, 주문 상태를 `PRODUCING → CONFIRMED`로 전환한다.
- 현재 생산 중인 시료 정보(주문 정보, 진행 상황 등)와 대기 중인 생산 큐 목록을 조회할 수 있다.

### 4.7 출고 처리
- `CONFIRMED` 상태 주문 목록을 표시하고, 특정 주문에 대해 출고를 실행한다.
- 출고 실행 시 주문 상태를 `RELEASE`로 전환한다.

## 5. 주문 상태 머신 (요약)

```
RESERVED --(거절)--------------------------> REJECTED               (종단, 모니터링 제외)
RESERVED --(승인, 재고 충분)---------------> CONFIRMED
RESERVED --(승인, 재고 부족)---------------> PRODUCING --(생산 완료)--> CONFIRMED
CONFIRMED --(출고)--------------------------> RELEASE
```

정의되지 않은 전이(예: `RELEASE` 상태 주문을 다시 승인)는 허용하지 않는다.

## 6. 데이터 모델 (개략)

| 엔티티 | 속성 |
|--------|------|
| Sample | 시료 ID, 이름, 평균 생산시간, 수율, 현재 재고 |
| Order | 주문번호, 시료 ID, 고객명, 주문 수량, 상태, 생성 시각 |

## 7. 비기능 요구사항

- **인터페이스**: 콘솔(CLI) 기반. 담당자가 메뉴 번호/명령을 직접 입력한다. 화면 레이아웃은 PDF의
  예시 UI를 참고하되 자유롭게 구성한다.
- **데이터 영속성**: JSON 파일 기반. 애플리케이션을 재시작해도 시료/주문 데이터가 유지되어야 한다.
- **아키텍처**: MVC(Model/View/Controller) + Repository 패턴으로 역할을 분리한다.
- **테스트**: GoogleTest/GoogleMock 기반 단위 테스트를 포함하며, TDD(Red-Green-Refactor)로 개발한다
  (`.claude/skills/test-driven-development/SKILL.md` 참고).
- **동시성**: 단일 프로세스 콘솔 앱으로 가정하며, 동시 접속/동시성 제어는 범위 밖이다.

## 8. 제약 및 가정

- 생산 라인은 단일 라인이며, 여러 라인을 두는 확장은 이번 범위에 포함하지 않는다.
- 사용자 인증/권한 분리는 구현하지 않는다 (역할은 개념적 구분일 뿐, 시스템 로그인 기능은 없음).
- 수율에 따른 실 생산량 계산에서 발생하는 잉여분(계산상 필요량보다 많이 생산되는 경우)은 재고로
  축적되는 것으로 가정한다.

## 9. Out of Scope

- 다중 생산 라인, 우선순위 기반 스케줄링(FIFO 외의 전략)
- 사용자 인증/권한 관리
- 외부 시스템(ERP, 이메일 알림 등) 연동
- 데이터베이스 기반 영속성 (JSON 파일로 확정)
